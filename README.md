# CLD — Web3 Cloud Storage

A blockchain-powered cloud storage demo built with Hardhat, Ethereum smart contracts, a Node.js Express backend, and a Web3 frontend.

## Project Overview

This project lets users upload files through a web UI, store file metadata on an Ethereum smart contract, and optionally pin the files to IPFS via Pinata. It includes:

- `contracts/CloudStorage.sol` — Solidity smart contract that stores per-user file metadata
- `server.js` — Express API for upload/download/delete, SIWE authentication, file hashing, local uploads, and optional IPFS pinning
- `js/app.js` — Frontend app UI for wallet connection, uploads, file browsing, and blockchain activity
- `js/web3.js` — Wallet integration, SIWE authentication, and contract interaction
- `js/crypto.js` — Client-side AES-256-GCM encryption using a wallet-derived key
- `scripts/deploy.js` — Hardhat deployment script that writes contract config to `js/config.js`

## Features

- Wallet connection via MetaMask / injected Ethereum provider
- Hardhat local network support
- File hashing with SHA-256
- Optional client-side AES-256-GCM encryption before upload
- File metadata stored on-chain with per-user access
- File download and deletion endpoints
- IPFS pinning support via Pinata API keys
- SIWE authentication with JWT for protected operations
- UI file categories, search, activity feed, and upload progress

## Prerequisites

- Node.js 18+ and npm
- MetaMask or another injected wallet
- `git` and basic command-line access

## Setup

1. Clone the repository

```bash
git clone https://github.com/AryaDevWeb/web3_cloud_storage.git
cd web3_cloud_storage
```

2. Install dependencies

```bash
npm install
```

3. Create a `.env` file (optional)

```bash
cp .env .env.local
```

Add any of the following variables:

```env
PORT=3000
JWT_SECRET=your_jwt_secret
PINATA_API_KEY=your_pinata_api_key
PINATA_SECRET_KEY=your_pinata_secret_key
```

If `JWT_SECRET` is not set, the server generates a random secret at runtime.

## Local Development

### 1. Start Hardhat node

```bash
npm run node
```

This launches a local Ethereum node at `http://127.0.0.1:8545`.

### 2. Deploy the smart contract

In a new terminal:

```bash
npm run deploy
```

This deploys `CloudStorage` and regenerates `js/config.js` with the deployed contract address and ABI.

### 3. Start the backend server

```bash
npm run dev
```

By default, the app serves static files from the project root and listens on `http://localhost:3000`.

### 4. Open the UI

Visit:

```text
http://localhost:3000
```

Connect your wallet and upload files.

## Smart Contract Details

### Contract: `CloudStorage`

Functions:

- `uploadFile(string _fileId, string _ipfsHash, string _fileName, uint256 _fileSize, bool _isEncrypted)`
- `getMyFiles() public view returns (File[] memory)`
- `getFileCount() public view returns (uint256)`
- `deleteFile(uint256 _index)`
- `getFile(uint256 _index) public view returns (File memory)`
- `renameFile(uint256 _index, string _newName)`

Events:

- `FileUploaded(address indexed owner, string fileId, string ipfsHash, string fileName, uint256 fileSize, uint256 uploadTime, bool isEncrypted)`
- `FileDeleted(address indexed owner, uint256 index, string fileId)`
- `FileRenamed(address indexed owner, uint256 index, string oldName, string newName)`

Metadata stored per user includes:

- fileId (SHA-256 hash)
- ipfsHash
- fileName
- fileSize
- uploadTime
- owner address
- encryption flag

## API Endpoints

### Public

- `GET /api/auth/nonce` — returns nonce for SIWE
- `POST /api/auth/verify` — verify SIWE signature and receive JWT
- `POST /api/upload` — upload a file to the server and compute SHA-256 file ID
- `GET /api/files/:fileId` — download file by ID
- `GET /api/health` — health status and IPFS availability

### Protected (JWT required)

- `DELETE /api/files/:fileId` — delete uploaded file

## Frontend Behavior

- `js/app.js` handles UI navigation, wallet login, file upload flow, category filters, and activity display.
- `js/web3.js` manages wallet connection, chain switching, SIWE signing, and contract calls.
- `js/crypto.js` derives an AES key from a wallet signature and encrypts/decrypts files in the browser.
- `js/config.js` is generated by the deployment script and contains network and contract metadata.

## Security Notes

- The server uses `helmet`, CORS, and rate limiting.
- File uploads are validated against dangerous extensions and path traversal.
- Optional Pinata IPFS pinning only runs when credentials are set.
- JWT tokens are issued for 24 hours after SIWE verification.

## Optional: Pinata / IPFS

If `PINATA_API_KEY` and `PINATA_SECRET_KEY` are provided, the backend pins uploaded files to IPFS and returns an IPFS hash with the upload response.

## Notes

- This project is built for local development and demonstration use.
- `uploads/` stores files on disk in hashed form.
- `js/ipfs.js` currently exists as an empty placeholder and is not required for base functionality.
